1. Purpose

The Logout feature must allow authenticated HMS users to end their current device session or all active device sessions. After logout, the active access token must be blocklisted so it cannot be reused to perform protected actions.

This feature is common for HMS enterprise and pharmacy users. It applies across roles, organisations, and supported dashboards.

2. Scope

In Scope

  • Current-device logout using POST /v1/api/logout.
  • Backend all-device logout support using POST /v1/api/logout/all.
  • Access-token blocklisting on successful logout.
  • Refresh-token removal for the current device.
  • Frontend confirmation before logout.
  • Redirect to the Login page after successful logout.
  • Common support for admin, staff, pharmacist, HMS patient, and pharmacy customer roles.

Out of Scope

  • Changing login, registration, or forgot-password behavior.
  • Changing role-based dashboard authorization.
  • Displaying a full device-management screen.

4. Product Requirements

  • Logout must be available from authenticated UI surfaces for all roles.
  • User must confirm before the UI calls the logout API.
  • If the user cancels confirmation, the session must continue without token changes.
  • Successful current-device logout must end only the current browser/device session.
  • Logout-all API remains available for backend/API clients but is not exposed as a UI option for now.
  • Successful logout must navigate the user to the Login page.
  • Protected APIs must reject blocklisted access tokens.

7. API Contract

7.1 Logout Current Device

MethodEndpointAuthDescription
POST/v1/api/logoutBearer access tokenBlocklist current access token and delete refresh token for the submitted device ID.

Request

POST /v1/api/logout?deviceId=web-chrome-01
Authorization: Bearer <accessToken>
Cookie: refresh_token=<refreshToken>

Success Response

200 OK
Set-Cookie: refresh_token=; Max-Age=0; Path=/; HttpOnly; SameSite=Lax

User logged out successfully

7.2 Logout All Devices Backend API

This API is documented for backend completeness, but the current UI confirmation dialog must not expose a logout-all-devices action.

MethodEndpointAuthDescription
POST/v1/api/logout/allBearer access tokenDelete all refresh tokens for the authenticated user and blocklist current access token.

Request

POST /v1/api/logout/all
Authorization: Bearer <accessToken>
Cookie: refresh_token=<refreshToken>

Success Response

200 OK
Set-Cookie: refresh_token=; Max-Age=0; Path=/; HttpOnly; SameSite=Lax

User logged out from all devices successfully

8. UI/UX Requirements

  • Logout button must be visible and reachable from authenticated dashboards.
  • Clicking Logout opens a modal confirmation dialog.
  • Dialog copy must make it clear that the session will end.
  • Primary action logs out the current device.
  • Logout-all-devices must not be shown in the confirmation dialog for now.
  • Cancel or closing the dialog must keep the current session active.
  • Dialog must be keyboard and screen-reader friendly using role="dialog" and aria-modal="true".
  • While logout is in progress, confirmation actions should be disabled to prevent duplicate requests.

9. Security Rules

  • Logout APIs require a bearer access token.
  • Access tokens added to token_blacklist must fail authentication on future protected API calls.
  • Refresh token cookie must be cleared after logout.
  • Current-device logout must require the device ID used during login.
  • Logout must remain allowed even when organisation write access is disabled, so users can exit safely.
  • Frontend must not keep stale local session data after logout.

11. Test Checklist

Backend

  • Current-device logout returns 200 with a valid bearer token and device ID.
  • Current-device logout inserts access token into token_blacklist.
  • Current-device logout removes only the matching device refresh token.
  • Logout-all removes all refresh tokens for the user.
  • Protected API request with blocklisted token returns unauthorized.
  • Logout APIs clear refresh_token cookie.

Frontend

  • Clicking Logout opens the confirmation dialog.
  • Cancel keeps the user on the current dashboard.
  • Confirm Logout calls current-device logout and redirects to Login.
  • Dialog works for admin, pharmacist, and customer dashboards.

12. Acceptance Criteria

  • Given an authenticated user, when they click Logout, then a confirmation dialog is shown.
  • Given the confirmation dialog is open, when the user clicks Cancel, then no logout API is called and the session continues.
  • Given the user confirms Logout, when the API succeeds, then the current token is blocklisted and the UI navigates to Login.
  • Given the confirmation dialog is open, then no logout-all-devices action is displayed for now.
  • Given a blocklisted token, when it is used on any protected API, then the action is rejected.
  • Given any supported HMS enterprise or pharmacy role, when logout is used, then the same common behavior applies.
