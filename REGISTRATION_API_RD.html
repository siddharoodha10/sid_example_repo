1. Purpose

The Registration API creates user accounts for HMS and pharmacy-only organisations. It must support public self-registration for patients and pharmacy customers, and admin-managed creation for staff and organisation users. Every created account must be linked to the correct active organisation and must follow role, uniqueness, activation, and database storage rules.

2. Scope

In Scope

  • Public PATIENT self-registration for HMS organisations.
  • Public CUSTOMER self-registration for pharmacy organisations.
  • Admin-created accounts for users inside the admin's organisation.
  • HMS role creation for ADMIN, DOCTOR, NURSE, PHARMACIST, LAB_STAFF, BILLING_STAFF, OTHER_STAFF, and PATIENT.
  • Pharmacy-only role creation for ADMIN, PHARMACIST, and CUSTOMER.
  • Organisation active-status validation before registration.
  • Admin active-status validation before admin-created user registration.
  • Uniqueness validation for username, phone number, email address, and staff employee ID where applicable.
  • First-login password-change flow for admin-created users.
  • Database placement rules for enterprise HMS users and pharmacy-only customers.
  • Development, testing, and product acceptance checklist.

Out of Scope

  • User login, token refresh, logout, and failed login lockout behavior.
  • Full UI page design beyond required validation and flow behavior.
  • Organisation onboarding and licence management.
  • Payment, billing, appointment, medicine order, or clinical workflows after registration.

4. Pre-Conditions

  • Organisation must exist and have org_status = ACTIVE before any account can be registered.
  • For admin-created users, the logged-in admin account must be active and belong to the organisation where the user is being created.
  • Admin-created users must not be allowed to provide or override another organisation ID in the request.
  • Public self-registration must resolve an active organisation from the request orgId or configured default organisation.
  • Role must be one of the supported role values for the organisation package type.
  • Password must be supplied and must pass password strength rules.
  • Required profile fields must be supplied for the selected role.

6. API Contract

6.1 Patient Self-Registration

MethodEndpointAuthPurpose
POST/v1/api/reg/patient/registerPublicCreate a public HMS patient account.

Request body:

{
  "username": "patient_john",
  "email": "john@example.com",
  "phone": "9876543210",
  "password": "Patient@123",
  "role": "PATIENT",
  "firstName": "John",
  "lastName": "Doe",
  "address": "Chennai, TN",
  "orgId": "HMS001",
  "dateOfBirth": "1990-01-01",
  "aadharNumber": "123412341234",
  "emergencyContactName": "Sita Devi",
  "emergencyContactNumber": "9123456789",
  "bloodGroup": "O+",
  "gender": "MALE"
}

Success response:

{
  "message": "Registration successful",
  "userId": 101,
  "username": "patient_john",
  "firstName": "John",
  "lastName": "Doe",
  "role": "PATIENT",
  "orgId": "HMS001"
}

Rules:

  • Role must be forced to PATIENT, even if another public role is sent.
  • Only active organisations can be used for patient self-registration.
  • Account should remain inactive until account confirmation is complete.
  • Record must be saved in users and patients.

6.8 First Login Password Change

MethodEndpointAuthPurpose
POST/v1/api/reg/first-login/change-passwordFirst-login session/tokenUpdate the temporary password after account creation and clear the first-password-change requirement.

Request body:

{
  "userId": 303,
  "role": "NURSE",
  "oldPassword": "Staff@123",
  "newPassword": "NewStaff@123",
  "confirmPassword": "NewStaff@123"
}

Success response:

{
  "message": "Password changed successfully. Account is ready to use.",
  "userId": 303,
  "username": "nurse_shyam",
  "role": "NURSE",
  "orgId": "HMS001",
  "isActive": true,
  "mustChangePassword": false
}

Rules:

  • This endpoint is required because the current implementation does not yet provide first-login password setup.
  • Applicable to admin-created admin, staff, HMS patient, and pharmacy customer accounts.
  • newPassword and confirmPassword must match and satisfy password strength rules.
  • Successful password change must encode the new password and clear mustChangePassword or the equivalent first-login flag.
  • For pending staff/admin accounts, successful password change must activate the account for normal application access.
  • For active admin-created patient/customer accounts, successful password change must keep the account active and allow normal application access.

Common Response Notes

  • All registration APIs return HTTP 201 Created on successful account creation.
  • First-login password change returns HTTP 200 OK after the password is updated successfully.
  • userId is sourced from users.user_id for HMS/admin/staff/patient users.
  • userId is sourced from pharmacyusers.pharmacy_user_id for standalone pharmacy customers.
  • Response must never include plain password, encoded password, OTP, reset token, lockout counters, or internal audit fields.
Admin-created APIs must ignore client-provided orgId and must use the logged-in admin's organisation. This prevents cross-organisation account creation.

9. Validation Rules

  • username is required and must be 3 to 50 characters.
  • firstName is required.
  • lastName is required for pharmacy customer registration.
  • email or emailAddress is required and must be valid email format.
  • phone or phoneNumber is required and must be 10 digits.
  • Password must contain minimum 8 characters, one uppercase letter, one lowercase letter, one number, and one special character.
  • orgId must refer to an active organisation when supplied.
  • Public patient registration must reject any non-PATIENT role.
  • Public customer registration must create only a CUSTOMER role.
  • Admin staff path role must be one of NURSE, PHARMACIST, LAB_STAFF, BILLING_STAFF, or OTHER_STAFF.
  • Admin customer creation is allowed only when the logged-in admin's organisation package type is PHARMACY_ONLY.
  • gender, when supplied, must be MALE, FEMALE, or OTHER.
  • bloodGroup, when supplied, should be one of A+, A-, B+, B-, AB+, AB-, O+, or O-.
  • aadharNumber, when supplied, must be 12 digits.
  • emergencyContactNumber, when supplied, must be 10 digits.
  • dateOfBirth, when supplied, must be in the past.
  • specialization and licenseNumber are required for doctor creation.
  • employeeId must be unique across all staff and doctor accounts inside the same organisation when supplied.
  • First-login password change must require old password, new password, confirm password, matching passwords, and password strength validation.

11. Security Rules

  • Password must never be logged or returned in API responses.
  • Password must be encoded before persistence.
  • Admin-created registration must use the authenticated admin identity from the security context.
  • Client-provided orgId must not be trusted for admin-created accounts.
  • Role values must be backend-enforced to prevent privilege escalation.
  • Registration must be transactional so base user and role profile are created together or rolled back together.
  • First-login password change must not allow normal application access until the temporary password is replaced where mustChangePassword is true.
  • Public registration endpoints should be rate limited to reduce automated fake account creation.
  • Duplicate-check responses should be useful but should not expose unnecessary private user details.

14. Acceptance Criteria

  • Given an active organisation, when a patient self-registers with valid unique details, then a PATIENT account is created under that organisation.
  • Given an inactive organisation, when any user attempts registration, then the registration is rejected.
  • Given an active admin, when the admin creates staff, then the staff account inherits the admin's organisation ID.
  • Given an admin from organisation A, when the admin attempts to create a user for organisation B, then the system prevents cross-organisation creation.
  • Given duplicate username, phone, email, or employee ID, when registration is submitted, then no account is created and a clear error is returned.
  • Given a pharmacy customer registration, when registration succeeds, then the account exists in pharmacyusers and not as a normal staff user in users.
  • Given a pharmacy-only admin, when the admin creates a customer, then the customer is created in pharmacyusers, inherits the admin's organisation ID, is active, and must change password on first login.
  • Given admin-created admin or staff registration, when the account is created, then the user is notified by email and must complete first-login password change before normal application access.
  • Given admin-created patient or customer registration, when the account is created, then the account is active but the first login requires password change before proceeding.
  • Given first-login password change succeeds, when the user logs in again, then the first-login password popup is no longer shown.

15. Resolved Product Decisions

  • Only admin, patient, and customer actors can initiate registration.
  • Admin-created accounts are organisation-scoped to the logged-in admin.
  • HMS organisations support full HMS roles; pharmacy-only organisations support admin, pharmacist, and customer roles.
  • Organisation package type is stored in the organisation table and controls allowed registration roles.
  • Enterprise HMS accounts are stored in users plus role profile tables.
  • Pharmacy customers are stored in pharmacyusers; pharmacy staff remain in users.
  • Public registration must not create active login-ready accounts until the account confirmation step is complete.
  • Admin-created admin and staff accounts require email notification and first-login password change before normal access.
  • Admin-created patient and customer accounts are active after creation but must change password on first login.
  • Employee ID uniqueness is global across all staff tables inside the same organisation.
  • A new first-login password-change endpoint is required because it does not currently exist.
  • A new admin-created pharmacy customer endpoint is required for pharmacy-only organisations.
